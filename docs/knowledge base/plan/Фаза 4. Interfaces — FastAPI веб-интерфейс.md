# Фаза 4: `interfaces/` — FastAPI веб-интерфейс

## Контекст

Ядро RAGv2 (core/, retriever/, agent/, eval/) готово и покрыто тестами. Агент вызывается программно (`agent.ask()`, `agent.ask_debug()`), но нет пользовательского слоя — доступа через браузер/чат. Требования проекта (§6) предписывают веб-приложение и Telegram-бот; план разбивает Фазу 4 на подфазы и в этой итерации закрывает **только FastAPI-часть**.

Цели фазы:
- Дать пользователю интерактивный чат с агентом (форма вопроса → ответ с источниками).
- Перенести `notebooks/debug.ipynb` в серверный HTML — полноценный debug dashboard (retrieval-таблица, LLM-вызовы, latency breakdown, JSON-экспорт).
- Дать инспектор чанков (поиск через retriever без агента) — §6.2.
- Дать ручную переиндексацию из UI — §4.
- Сохранить короткую память между сообщениями одного пользователя через HTTP-cookie + существующий `MemorySaver`.

Отложено за пределы фазы (по согласованию): Telegram-бот, Langfuse, HITL для `manage_note`, Settings UI для рантайм-конфига, аутентификация, переход на SqliteSaver, streaming-ответы.

## Переиспользуемый API (не модифицируется)

- `agent.ask(question, thread_id=None) → AgentResponse` — `agent/__init__.py`
- `agent.ask_debug(question, thread_id=None) → (AgentResponse, DebugTrace)` — `agent/__init__.py`
- `agent.get_mermaid() → str` — для диаграммы графа в debug-странице
- `agent.tracer.DebugTrace.to_dict()` — сериализация трейса в шаблон
- `core.types.AgentResponse` — база для `AskResponse` DTO
- `core.config.get_config()` — `cfg.eval.chunk_preview_len`, `cfg.agent.max_iterations`
- `retriever.search.search(query, bm25_terms=None)` — для chunk-инспектора
- `retriever.indexer.run_indexing(force=False) → dict` — для /api/reindex

## Ключевые решения

### Сессии (`thread_id`)
- Cookie `rag_thread_id` (HttpOnly, SameSite=Lax, TTL 24ч). Генерится через `uuid4()` при первом запросе.
- `AskRequest.thread_id` в body переопределяет cookie (debug-сценарий).
- Кнопка «Сбросить сессию» → `POST /api/thread/reset` → новый UUID в cookie.
- `MemorySaver` остаётся — known limitation: история теряется при рестарте процесса.

### Обработка ошибок (§1.1)
- Кастомные: `LlmUnavailableError`, `SearchBackendError`, `ReindexError`.
- Глобальные хендлеры в `errors.py`: 503 при недоступности LLM, 500 catch-all с рендером `errors/5xx.html` для HTML-страниц.
- `AgentResponse.has_answer=False` — НЕ ошибка: HTTP 200 + жёлтая плашка-баннер в UI.

### Reindex (sync работа внутри async эндпоинта)
- `reindex_manager.py`: глобальный `_current_job: ReindexStatus | None` под `threading.Lock()`.
- `start(force)` → проверяет lock, стартует `threading.Thread(daemon=True, target=_run_job)`. Если уже `running` → 409.
- `_run_job(force)` вызывает `run_indexing(force)`, обновляет статус (`running` → `done`/`error`).
- `/admin` страница: JS `setInterval` 5сек опрашивает `GET /api/reindex/status` пока `status=="running"`.

### Debug dashboard
- `POST /debug` server-rendered (без SPA): форма → вызов `ask_debug()` → рендер `debug.html` с `trace` и `response`.
- Четыре фрагмента-шаблона повторяют секции `notebooks/debug.ipynb`:
  1. `_trace_summary.html` — вопрос, thread_id, latency, iterations, chunks_used, confidence.
  2. `_trace_retrieval.html` — по каждому `ToolEvent(search_knowledge_base)` таблица `retrieved_docs` + CSS-бар score.
  3. `_trace_llm.html` — раскрывающиеся `<details>` на каждый LLMEvent: node, latency, messages_in (truncate 800), tool_calls, response.
  4. `_trace_latency.html` — LLM/Tools/Overhead с процентами и inline-бар.
- Плюс JSON-export (`<pre>` + Copy) и Mermaid-диаграмма через CDN (`mermaid.min.js`).

### Chunk-инспектор
- `GET /chunks?q=...&bm25_terms=...&top_k=10` — server-rendered, вызывает `retriever.search.search()` напрямую.
- `top_k` обрезаем до 50; `use_reranking` в MVP игнорируем (TODO).
- Показываем heading_hierarchy, type, tags, created, превью `content[:cfg.eval.chunk_preview_len]`.
