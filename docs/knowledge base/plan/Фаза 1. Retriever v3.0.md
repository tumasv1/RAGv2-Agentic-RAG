# Parent-Child чанкинг

## Context

Сейчас чанкинг делает малые чанки (MHTS → RCTS, chunk_size=800). Поиск возвращает эти же малые чанки в LLM — кусок документа «вылетает» если один-два соседних чанка не попали в топ. Цель: оставить поиск на малых чанках (высокая точность), но передавать в LLM крупного родителя (до 2000 символов) — полный контекст при той же точности. 74% документов vault'а короче 2000 chars → большинство файлов станут одним parent'ом.

## Архитектура

**Одна коллекция `obsidian_notes`, два уровня:**
- `metadata.kind="child"` — малые чанки, ищутся (dense + BM25 + reranker).
- `metadata.kind="parent"` — крупные чанки, НЕ ищутся. Достаются по `parent_id` → отдаются в LLM (дедуп). При отправке в LLM добавляются метданнные (source, created, source_part, type, tegs).

**Ключевой принцип (по уточнению пользователя):** children строятся **из текста parent'а**, а не из исходных секций. Связь child→parent явная, сопоставление по offset'ам не нужно.

## Алгоритм chunk_file()

1. `_extract_frontmatter` + `_normalize_wikilinks` — без изменений.
2. `MHTS(body)` → `sections: list[Document]`.
3. **Построение parents (greedy grouping, новая функция `_build_parents`):**
   - Предельный размер — `cfg.ingest.parent_chunk_size` (новый параметр, дефолт 2000).
   - Идём по секциям с буфером `current_buffer`.
   - Если одна секция > `parent_chunk_size` → сначала флашим буфер как parent, затем секцию режем `RCTS(parent_chunk_size, 0)` (**без overlap** — поиск по parents не идёт), каждый кусок = отдельный parent.
   - Если `current_buffer + section > parent_chunk_size` → закрываем буфер как parent, начинаем новый с текущей секции.
   - Иначе → дописываем секцию в буфер.
   - По окончании флашим остаток.
   - Parent'у присваиваем: `parent_index` (0..N-1), `parent_total=N` (для метаданных — не для prefix'а).
   - `chunk_id` parent'а: `MD5(f"{file_path}:parent:{parent_index}")`.
4. **Построение children — из текста parent'а:**
   - Для каждого parent'а: `MHTS(parent.text)` → re-sections, затем `RCTS(cfg.chunk_size, cfg.chunk_overlap).split_documents(re_sections)` — логика как сейчас.
   - У каждого child'а: `parent_id = parent.chunk_id` (явно, без матчинга), `parent_file_name = file_path.name` (для дебага по запросу пользователя), `file_path`, `file_name`, `type`, `created`, `tags` из метаданных parent.
   - `chunk_id` child'а: `MD5(f"{file_path}:child:{global_index}")` — global_index сквозной по всем parent'ам файла.
5. **Prefix-построение (упрощённый вариант для children, полный для parents):**

   - **Child prefix** (упрощённая версия `_build_context_prefix`):
     ```
     Файл: {file_name}
     Тип: {type}                  # только если non-empty
     Теги: {tag1, tag2, …}        # только если tags non-empty
     ---
     ```
     — префикс добавляется к `page_content` при формировании чанка, чтобы ембединг шел с учетом префикса.
6. Возвращаем tuple `(children, parents)` — каждый элемент `(text_with_prefix, ChunkMetadata)`.

**Замечание:**    -Parent prefix добавляется в `agent/tools.py:_format_chunk` в момент отдачи LLM. В `page_content` родителя не добавляем, чтобы он не попал в child-chunk.
   Формат Parent prefix
     ```
     [N]                          # порядковый номер в контексте
     source: {file_name}
     created: {created}           # только если non-empty
     source_part: {i+1}/{N}       # только если parent_total > 1
     type: {type}                 # только если non-empty
     tegs: {tag1, tag2, …}        # только если tags non-empty
     ---
     ```

### `retriever/search.py`

Разделить логику:

```python
def _search_children(query, bm25_terms) -> list[SearchResult]:
    # текущая логика query_points + prefetch + reranker
    # ДОБАВИТЬ: query_filter = Filter(must=[FieldCondition(key="metadata.kind", match=MatchValue(value="child"))])

def _dedup_and_fetch_parents(children: list[SearchResult]) -> list[SearchResult]:
    seen: dict[str, float] = {}           # parent_id → max child-score
    for r in children:
        pid = r.metadata.parent_id
        if pid:
            seen[pid] = max(seen.get(pid, -1e9), r.score)
    if not seen:
        return []
    points = store.client.retrieve(
        collection_name=store.collection_name,
        ids=list(seen.keys()),
        with_payload=True,
    )
    parents = []
    for p in points:
        sr = _point_to_result(p)
        if sr:
            sr = SearchResult(content=sr.content, metadata=sr.metadata,
                              score=seen[sr.metadata.chunk_id])
            parents.append(sr)
    parents.sort(key=lambda r: r.score, reverse=True)
    return parents[:cfg.max_chunks]

def search(query, bm25_terms=None) -> list[SearchResult]:
    return _dedup_and_fetch_parents(_search_children(query, bm25_terms))

def search_with_detail(query, bm25_terms=None) -> tuple[list[SearchResult], list[SearchResult]]:
    """Возвращает (parents, children) — для eval pipeline."""
    children = _search_children(query, bm25_terms)
    return _dedup_and_fetch_parents(children), children
```

Аналогичные изменения в `make_search_fn()` (для `eval/compare_splitters.py`) — возвращать пару `(search_fn, search_detail_fn)`.

CLI `__main__`: показывать parents + под каждым список child'ов, которые привели к нему.

### Общий форматтер parent prefix

Выносим в новый модуль `retriever/formatting.py` (общее место для agent и eval):

```python
def build_parent_prefix(index: int, meta: ChunkMetadata) -> str:
    """
    Собирает prefix parent-чанка для подачи в LLM или показа в отчёте.
    `index` — порядковый номер в контексте (1-based).
    """
    lines = [f"[{index}]", f"source: {meta.file_name}"]
    if meta.created:
        lines.append(f"created: {meta.created}")
    if meta.parent_total and meta.parent_total > 1:
        lines.append(f"source_part: {meta.parent_index + 1}/{meta.parent_total}")
    if meta.type:
        lines.append(f"type: {meta.type}")
    if meta.tags:
        lines.append(f"tegs: {', '.join(meta.tags)}")
    return "\n".join(lines) + "\n---"
```

Переиспользуется из `agent/tools._format_chunk` и `eval/runner.py` при построении `ChunkInfo.preview` для parent-чанков.

### `agent/tools.py:_format_chunk`

```python
from retriever.formatting import build_parent_prefix

def _format_chunk(i: int, r: SearchResult) -> str:
    prefix = build_parent_prefix(i, r.metadata)
    return f"{prefix}\n{r.content.strip()}"
```

Новый разделитель между чанками `"\n\n======\n\n"`, чтобы не путаться с разделителем `---` внутри prefix'а.

## Риски и заметки

- **Коллизии chunk_id при миграции:** префиксирование `child:`/`parent:` в `_generate_chunk_id` изменит все id → обязательна `--force` переиндексация.
- **Storage:** parents тоже эмбеддятся (Qdrant требует vector на insert). Лишние ~20-30% памяти — приемлемо.
- **Reranker** применяется на уровне **children** (до dedup). Parent'у назначается `score = max(score of его children в результатах)`. Реранкер точнее на коротких child-чанках — это осознанный выбор.
- **Короткие файлы (≤2000 chars, 74% vault):** parent = весь файл одним чанком, `parent_total=1`, `source_part` не показывается. Children этого файла — как и раньше, MHTS+RCTS его секций.
- **eval pipeline:** две таблицы позволяют видеть children-scores и калибровать `dense_score_threshold` глядя на тот уровень, на котором реально идёт поиск.

## Верификация

1. **chunker CLI:** `python -m retriever.chunker /путь/к/длинной/заметке.md` — проверяем что children корректно указывают на parent_id, `parent_total > 1` для длинных файлов, `parent_file_name` совпадает с file_name.
2. **search CLI:** `python -m retriever.search "Галаева"` — ожидаем parent (целый файл «Галаева Елена.md»), а не несколько child-фрагментов.
3. **E2E через web:** `POST /api/ask` на «Кто такая Галаева Елена?» — has_answer=true, sources = 1 файл, формат parent-чанка в debug-трейсе совпадает со спецификацией.
4. **/debug:** kind=parent, prefix содержит `source:`, `source_part:` если парент-частей > 1, `tegs:` если есть теги.
5. **RAGAS:** ожидаем улучшение `context_recall` (меньше потерь соседнего текста), `faithfulness` не хуже baseline.
6. **eval report:** в каждом кейсе две таблицы parents/children; parent-preview содержит полный prefix, children-preview — упрощённый.
