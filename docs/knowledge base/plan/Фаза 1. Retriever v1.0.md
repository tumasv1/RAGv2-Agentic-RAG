## Контекст

Фаза 0 (core/) реализована: конфиг, LLM-клиент, типы данных. Теперь строим модуль `retriever/` — сердце RAG-системы. Он отвечает за три задачи: разбить markdown-заметки на чанки с метаданными, сохранить их в Qdrant, и найти релевантные чанки по запросу.

Модуль зависит только от `core/` и не зависит от agent/, eval/, interfaces/.

Что берём из RAG v1, а что делаем иначе:

- ✅ Переиспользуем: регулярки для frontmatter и wikilinks, стабильные ID чанков (MD5), mtime-индексацию
- ✅ Переиспользуем: список исключённых папок (шаблоны, вложения, ingest-логи)
- 🔄 Улучшаем: ChromaDB → Qdrant с нативным гибридным поиском (dense + BM25 + RRF)
- 🔄 Улучшаем: извлечение heading_hierarchy (хлебные крошки заголовков для каждого чанка)
- ❌ Не берём: сложный date-section splitting (избыточен), RunReport dataclass (тяжеловесно)

## Шаг 1: `retriever/embeddings.py`

**Файл:** `/Users/mikhail/Projects/RAGv2/retriever/embeddings.py`

Синглтон для модели эмбеддингов. Модель загружается ~10 сек и занимает ~1.5 ГБ RAM, поэтому создаём один раз.

### Ключевая функция

python

```python
from langchain_huggingface import HuggingFaceEmbeddings
from core.config import get_config

_embeddings: HuggingFaceEmbeddings | None = None

def get_embeddings() -> HuggingFaceEmbeddings:
    """Синглтон модели эмбеддингов. Первый вызов загружает модель."""
    global _embeddings
    if _embeddings is None:
        cfg = get_config().embeddings
        _embeddings = HuggingFaceEmbeddings(
            model_name=cfg.model_name,
            model_kwargs={"device": cfg.device},
            encode_kwargs={
                "normalize_embeddings": cfg.normalize,
                "prompt": "passage: ",       # E5 требует префикс для документов
            },
            query_encode_kwargs={
                "normalize_embeddings": cfg.normalize,
                "prompt": "query: ",         # E5 требует префикс для запросов
            },
        )
    return _embeddings
```

> **Почему `prompt: "passage: "` / `"query: "`?** Модель `intfloat/multilingual-e5-large` обучена с этими префиксами. Без них качество эмбеддингов падает. Параметры `encode_kwargs` и `query_encode_kwargs` проверены в исходниках `langchain-huggingface` (строка 54, поддерживается).

### CLI: `python -m retriever.embeddings`

- Загружает модель
- Эмбеддит тестовый текст
- Печатает длину вектора и время загрузки

## Шаг 2: `retriever/chunker.py`

**Файл:** `/Users/mikhail/Projects/RAGv2/retriever/chunker.py`

Принимает путь к `.md` файлу → возвращает список `(текст_чанка, ChunkMetadata)`.

### Подход: MarkdownHeaderTextSplitter + RecursiveCharacterTextSplitter

Двухэтапный чанкинг — стандартный паттерн LangChain:

1. **MarkdownHeaderTextSplitter** — разбивает текст по заголовкам (H1–H4). Каждая секция получает метаданные `{"Header 1": "...", "Header 2": "...", ...}` — это наша `heading_hierarchy` бесплатно.
2. **RecursiveCharacterTextSplitter** — дорезает большие секции до `chunk_size`. Маленькие секции остаются как есть.

> **Почему не один RecursiveCharacterTextSplitter с кастомной картой заголовков?** Связка MHTS+RCTS проще: не нужен хак с `str.find()` для позиций чанков, heading_hierarchy достаётся из метаданных автоматически, меньше кастомного кода.

### Алгоритм `chunk_file(file_path)`

1. Прочитать файл
2. Извлечь frontmatter (regex `^---\n...\n---`) → dict с полями
3. Убрать frontmatter из текста
4. Нормализовать wikilinks: `[[target|alias]]` → `alias`, `[[target]]` → `target`
5. **MarkdownHeaderTextSplitter** → список Document с метаданными заголовков
6. **RecursiveCharacterTextSplitter** → дорезать большие секции
7. Сгенерировать `chunk_id = MD5("file_path:index")`
8. Собрать `ChunkMetadata` из метаданных Document + frontmatter

### Ключевые функции

python

```python
def chunk_file(file_path: Path) -> list[tuple[str, ChunkMetadata]]
def _extract_frontmatter(text: str) -> tuple[dict, str]
def _normalize_wikilinks(text: str) -> str
def _generate_chunk_id(file_path: str, index: int) -> str
```

### Настройка сплиттеров

python

```python
# Шаг 1: разбиение по заголовкам → секции с метаданными
HEADERS_TO_SPLIT = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
    ("####", "Header 4"),
]
md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=HEADERS_TO_SPLIT,
    strip_headers=False,  # оставляем заголовки в тексте — полезно для контекста
)

# Шаг 2: дорезаем большие секции по размеру
SEPARATORS = [
    "\n```",           # блоки кода
    "\n> [!",          # callouts (Obsidian)
    "\n\n",            # параграфы
    "\n",              # строки
    " ",               # слова
]
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=cfg.chunk_size,
    chunk_overlap=cfg.chunk_overlap,
    separators=SEPARATORS,
)
```

> **Заголовков нет в SEPARATORS** — разбиение по заголовкам уже сделано на первом шаге (MHTS). RCTS режет только внутри секций.

### Извлечение heading_hierarchy из метаданных Document

python

```python
# MarkdownHeaderTextSplitter кладёт заголовки в doc.metadata:
# {"Header 1": "Проекты", "Header 2": "Работа", "Header 3": "Задачи"}
# Превращаем в heading_hierarchy и section_header:

heading_hierarchy = [
    doc.metadata[key]
    for key in ("Header 1", "Header 2", "Header 3", "Header 4")
    if key in doc.metadata
]
section_header = heading_hierarchy[-1] if heading_hierarchy else ""
```

### Обработка frontmatter

- `type`: может быть строкой или списком → берём первый элемент если список
- `created`: строка в формате `YYYY-MM-DD` или пустая
- `tags`: список или строка → нормализуем в список
- Остальные поля → `extra_metadata`

### CLI: `python -m retriever.chunker /path/to/file.md`

- Показывает каждый чанк с метаданными (chunk_id, section_header, heading_hierarchy)
- Печатает число чанков и средний размер

## Шаг 3: `retriever/indexer.py`

**Файл:** `/Users/mikhail/Projects/RAGv2/retriever/indexer.py`

Инкрементальная индексация: сканирует vault, сравнивает с сохранённым состоянием по mtime, чанкует изменённые файлы, сохраняет в Qdrant.

### Синглтон Qdrant Store

python

```python
from langchain_qdrant import QdrantVectorStore, RetrievalMode, FastEmbedSparse

def get_qdrant_store() -> QdrantVectorStore:
    """Синглтон QdrantVectorStore в гибридном режиме (dense + BM25 + RRF)."""
    ...
    _qdrant_store = QdrantVectorStore.construct_instance(
        embedding=get_embeddings(),
        sparse_embedding=FastEmbedSparse("Qdrant/bm25"),
        retrieval_mode=RetrievalMode.HYBRID,
        client_options={"path": str(qdrant_path)},
        collection_name=cfg.qdrant.collection_name,
    )
```

> **Почему `construct_instance`, а не конструктор напрямую?** Этот classmethod (строка 891 `qdrant.py`) автоматически создаёт коллекцию с правильной конфигурацией векторов (dense + sparse) при первом запуске. Обычный конструктор требует, чтобы коллекция уже существовала.

### index_state.json

Хранится в `{qdrant.path}/index_state.json` — рядом с данными Qdrant. Если удалить папку qdrant_data/, состояние тоже удалится — не будет рассинхронизации.

Формат: `{"абсолютный_путь": mtime_float, ...}`

### Ключевые функции

python

```python
def get_qdrant_store() -> QdrantVectorStore   # синглтон Qdrant-хранилища
def run_indexing(force: bool = False) -> dict  # главная точка входа
def _scan_vault() -> dict[str, float]          # сканирование vault → {path: mtime}
def _load_state() -> dict[str, float]          # загрузка index_state.json
def _save_state(state) -> None                 # сохранение index_state.json
def _delete_chunks_for_files(store, paths)     # удаление чанков из Qdrant по file_path
def _index_files(store, paths) -> int          # чанкинг + upsert → кол-во чанков
```

### Алгоритм `run_indexing(force)`

1. Сканируем vault → `{path: mtime}`
2. Загружаем `index_state.json` (или пустой dict если force=True)
3. Сравниваем: new (нет в state), changed (mtime выше), deleted (нет в vault)
4. Для changed + deleted: удаляем старые чанки через Qdrant filter по `metadata.file_path`
5. Для new + changed: чанкуем через `chunk_file()` → `store.add_texts(texts, metadatas, ids)`
6. Сохраняем обновлённый `index_state.json`
7. Печатаем итоги: добавлено, обновлено, удалено, без изменений

### Удаление чанков

Используем filter-based deletion через Qdrant API (а не `store.delete(ids=[...])`) — не нужно хранить маппинг файл→ID чанков:

python

```python
from qdrant_client import models as qmodels

store.client.delete(
    collection_name=store.collection_name,
    points_selector=qmodels.FilterSelector(
        filter=qmodels.Filter(must=[
            qmodels.FieldCondition(
                key="metadata.file_path",
                match=qmodels.MatchValue(value=file_path),
            )
        ])
    ),
)
```

### Метаданные в Qdrant

`store.add_texts(texts, metadatas, ids)` — metadatas передаём как `[chunk_meta.model_dump(), ...]`. Qdrant сохраняет это в payload как `{"page_content": text, "metadata": {...}}`. При поиске `doc.metadata` вернёт тот же dict.

### CLI

- `python -m retriever.indexer` — инкрементальная индексация
- `python -m retriever.indexer --force` — полная переиндексация

## Шаг 4: `retriever/search.py`

**Файл:** `/Users/mikhail/Projects/RAGv2/retriever/search.py`

Обёртка над гибридным поиском Qdrant → возвращает `list[SearchResult]`.

### Ключевая функция

python

```python
from retriever.indexer import get_qdrant_store

def search(query: str) -> list[SearchResult]:
    """
    Гибридный поиск: dense + BM25 + RRF.

    1. similarity_search_with_score(query, k=fetch_k, score_threshold)
    2. Конвертация Document → SearchResult (с реконструкцией ChunkMetadata)
    3. Ограничение до max_chunks результатов
    """
```

> **Почему `search.py` импортирует `get_qdrant_store` из `indexer.py`?** Это тот же синглтон — и indexer, и search работают с одним Qdrant-хранилищем. Альтернатива — вынести в отдельный `_store.py`, но это добавит файл без реальной пользы.

### Параметры поиска

- `k=cfg.fetch_k` (40) — сколько кандидатов забрать из Qdrant (из обеих веток: dense и sparse)
- `score_threshold=cfg.score_threshold` (0.72) — отсечение по score (обрабатывается Qdrant)
- Финальное ограничение: `results[:cfg.max_chunks]` (10)

### Конвертация Document → SearchResult

python

```python
for doc, score in results:
    meta_dict = doc.metadata.copy()
    meta_dict.pop("_id", None)              # служебное поле Qdrant
    meta_dict.pop("_collection_name", None)  # служебное поле Qdrant
    chunk_meta = ChunkMetadata(**meta_dict)
    search_results.append(SearchResult(content=doc.page_content, metadata=chunk_meta, score=score))
```

### CLI: `python -m retriever.search "запрос"`

- Показывает результаты с score, именем файла, секцией, превью текста

## Возможные проблемы

1. **Первая индексация ~480 файлов на CPU** займёт 15-30 минут (E5-large на CPU). Добавим прогресс-вывод каждые 10 файлов.
2. **Qdrant embedded mode** захватывает file lock — одновременно индексировать и искать из разных процессов нельзя. В рамках проекта (один пользователь) это не проблема.
3. **E5-префиксы** (`query:` / `passage:`) критичны для качества. Настраиваем через `encode_kwargs` / `query_encode_kwargs` в embeddings.py — чанкер и поиск не должны об этом знать.
