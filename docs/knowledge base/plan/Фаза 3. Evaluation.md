# Модуль 3: eval/ — Оценка качества

## Контекст

Фазы 0–2 завершены: core/, retriever/ (2651 чанк, гибридный поиск), agent/ (ReAct-граф). Следующий шаг — eval/ для измерения качества retrieval и подбора параметров (пороги, реранкер, стратегии чанкинга) через RAGAS-бенчмарки. Модуль eval/ тестирует **retriever + LLM напрямую**, минуя граф агента — это изолирует оценку retrieval-качества от логики оркестрации.

## Структура файлов

```
eval/
├── __init__.py              # реэкспорт run_evaluation
├── golden_set.yaml          # уже есть (18 тест-кейсов)
├── runner.py                # ядро: прогон golden set через search + LLM, сборка Dataset
├── metrics.py               # обёртки RAGAS, вычисление метрик
├── report.py                # генерация markdown-отчёта (секции, светофоры)
├── eval_ragas.py            # CLI-точка входа: python -m eval.eval_ragas
└── compare_splitters.py     # CLI: сравнение стратегий чанкинга
```

**Почему 3 файла вместо одного eval_ragas.py из High-level plan:**
В RAG v1 всё было в одном файле (400 строк) — это работало, но при добавлении compare_splitters код дублируется. Разделяем на runner (прогон), metrics (RAGAS), report (вывод) — compare_splitters переиспользует runner и metrics с другой коллекцией.

## Файл 1: `eval/runner.py` — Прогон golden set

Ядро модуля: загружает тест-кейсы, для каждого вызывает search + LLM, собирает данные для RAGAS.

### Функции

```python
def load_golden_set(path: Path | None = None, n: int | None = None) -> list[dict]
```
Загружает golden_set.yaml, возвращает первые `n` кейсов (или все).

```python
def generate_answer(question: str, contexts: list[str]) -> LLMAnswer
```
Вызывает `get_llm().with_structured_output(LLMAnswer)` — структурированный ответ по контексту. Промпт минимальный: "Ответь на вопрос ТОЛЬКО на основе контекста. Формат: answer, sources, has_answer, confidence."

`LLMAnswer` — Pydantic-модель (в этом же файле):
```python
class LLMAnswer(BaseModel):
    answer: str
    sources: list[str] = []
    has_answer: bool
    confidence: float = Field(ge=0.0, le=1.0)
```

```python
def run_golden_set(cases: list[dict]) -> EvalDataset
```
Для каждого кейса:
1. `search(question)` → `list[SearchResult]`
2. `generate_answer(question, [r.content for r in results])` → `LLMAnswer`
3. Собирает в `EvalDataset`

`EvalDataset` — dataclass с полями:
- `questions: list[str]`
- `answers: list[str]`
- `contexts: list[list[str]]` — тексты чанков
- `ground_truths: list[str]` — reference_answer из golden set
- `chunks_detail: list[list[ChunkInfo]]` — source, score, preview для отчёта
- `has_answers: list[bool]`
- `cases: list[dict]` — исходные кейсы (для id, type, weight)

Метод `to_ragas_dataset() → Dataset` конвертирует в формат RAGAS:
```python
Dataset.from_dict({
    "question": self.questions,
    "answer": self.answers,
    "contexts": self.contexts,
    "ground_truth": self.ground_truths,
})
```

### Ключевое решение: search() vs search с параметром collection

Для eval_ragas.py — вызываем обычный `search()` из retriever (текущая коллекция).
Для compare_splitters — нужен поиск по временной коллекции. Добавим параметр `collection_name: str | None = None` в `search()` (если None — используется дефолтная коллекция). Альтернатива — создавать отдельный QdrantVectorStore для каждой коллекции внутри compare_splitters.

**Выбор:** параметр `search_fn` в `run_golden_set()` — позволяет передать кастомную функцию поиска (для compare_splitters создадим обёртку над временной коллекцией). Не трогаем retriever/search.py.

```python
def run_golden_set(
    cases: list[dict],
    search_fn: Callable[[str], list[SearchResult]] | None = None,  # дефолт: retriever.search.search
) -> EvalDataset
```

## Файл 2: `eval/metrics.py` — RAGAS-метрики

```python
def setup_ragas_llm() -> LangchainLLMWrapper
```
Оборачивает `get_llm()` из core в `LangchainLLMWrapper`.

```python
def setup_ragas_embeddings() -> LangchainEmbeddingsWrapper
```
Оборачивает `get_embeddings()` из retriever в `LangchainEmbeddingsWrapper`.

```python
def compute_metrics(dataset: Dataset) -> dict
```
Вызывает `ragas.evaluate()` с 4 метриками: faithfulness, answer_relevancy, context_precision, context_recall. Возвращает результат (dict-like объект с per-sample scores).

Импорты RAGAS 0.2.x:
```python
from ragas import evaluate
from ragas.llms import LangchainLLMWrapper
from ragas.embeddings import LangchainEmbeddingsWrapper
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall
```

## Файл 3: `eval/report.py` — Markdown-отчёт

Паттерн из RAG v1: модульные секции, каждая — отдельная функция.

### Секции отчёта

1. **Настройки pipeline** — max_chunks, fetch_k, thresholds, use_reranking (из config)
2. **Средние метрики** — 4 RAGAS-метрики, средние по всем кейсам
3. **Скоры по кейсам** — таблица: id, вопрос, 4 метрики, кол-во чанков, has_answer, светофор
4. **Ответы LLM** — полный текст ответа модели на каждый вопрос
5. **Диагностика context_recall** — кейсы ниже порога (0.7), найденные vs ожидаемые источники
6. **Контекст (чанки)** — таблица чанков с source, score, preview для каждого вопроса

### Функции

```python
def write_report(result: dict, eval_data: EvalDataset, output_path: Path | None = None) -> Path
```
Собирает все секции, записывает в `reports/ragas_report_YYYY-MM-DD.md` (append mode). Возвращает путь.

Вспомогательные: `_fmt(val)`, `_traffic_light(scores)` — из RAG v1.

### Светофоры
- 🟢 все ≥ 0.70
- 🟡 мин ≥ 0.40
- 🔴 мин < 0.40
- ⚪ нет данных

## Файл 4: `eval/eval_ragas.py` — CLI

Точка входа: `python -m eval.eval_ragas [--samples N]`

```python
def main(n_samples: int | None = None) -> None:
    cases = load_golden_set(n=n_samples)
    eval_data = run_golden_set(cases)
    dataset = eval_data.to_ragas_dataset()
    result = compute_metrics(dataset)
    report_path = write_report(result, eval_data)
    print(f"Отчёт: {report_path}")
```

CLI через `argparse`:
- `--samples N` — количество кейсов (дефолт: все)

## Файл 5: `eval/compare_splitters.py` — Сравнение стратегий

### Стратегии для сравнения

1. **baseline** (текущая) — MHTS + RCTS, chunk_size=1700, overlap=200
2. **mhts_only** — только MarkdownHeaderTextSplitter (без дорезки RCTS)
3. **small_chunks** — MHTS + RCTS, chunk_size=800, overlap=100
4. **large_chunks** — MHTS + RCTS, chunk_size=2500, overlap=300

Parent-Child — сложная стратегия, требует изменений в search.py (MultiVectorRetriever). Оставляем на потом, не включаем в первую версию.

### Реализация

Каждая стратегия — dataclass:
```python
@dataclass
class SplitterStrategy:
    name: str
    chunk_size: int
    chunk_overlap: int
    use_mhts: bool = True       # MarkdownHeaderTextSplitter
    use_rcts: bool = True       # RecursiveCharacterTextSplitter
```

Для каждой стратегии:
1. Создать временную коллекцию в Qdrant (`tmp_{strategy.name}`)
2. Проиндексировать vault этой стратегией (модифицируя параметры chunker)
3. Создать `search_fn` — обёртку для поиска по этой коллекции
4. Прогнать golden set через `run_golden_set(cases, search_fn=search_fn)`
5. Вычислить RAGAS-метрики
6. Удалить временную коллекцию

### Передача параметров чанкинга

Не модифицируем `chunker.py`. Вместо этого — временно подменяем значения в конфиге:
```python
cfg = get_config()
cfg.ingest.chunk_size = strategy.chunk_size
cfg.ingest.chunk_overlap = strategy.chunk_overlap
```
А для `use_rcts=False` — создаём обёртку `chunk_file_mhts_only()` в compare_splitters (вызывает только MHTS из chunker, без RCTS).

### Работа с временными коллекциями

Используем `QdrantClient` напрямую (из qdrant-client):
```python
from qdrant_client import QdrantClient
client = QdrantClient(path=str(qdrant_path))
# create_collection / delete_collection
```

Для индексации — создаём `QdrantVectorStore` с кастомным `collection_name`, переиспользуя логику из `indexer._index_files()`.

### Отчёт

Сводная таблица:
```
| Стратегия    | chunks | Faith | AnswRel | CtxPrec | CtxRec | Итог |
|--------------|--------|-------|---------|---------|--------|------|
| baseline     | 2651   | 0.85  | 0.90    | 0.75    | 0.60   | 🟡   |
| mhts_only    | 1800   | 0.80  | 0.88    | 0.70    | 0.55   | 🟡   |
| small_chunks | 4200   | 0.82  | 0.87    | 0.80    | 0.65   | 🟡   |
```

CLI: `python -m eval.compare_splitters [--samples N] [--strategies baseline,small_chunks]`

### Время выполнения

~9 мин на индексацию одной стратегии (CPU). 4 стратегии ≈ 36 мин + RAGAS eval. Выводим прогресс для каждого этапа.
