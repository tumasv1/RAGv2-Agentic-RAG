## Контекст

Фазы 0 (core/) и 1 (retriever/) завершены. Есть работающий гибридный поиск `search(query, bm25_terms)`, LLM-клиент `get_llm()`, типы данных `AgentResponse`. Теперь нужен LangGraph-граф, который оркестрирует поиск по базе знаний и формирует ответ пользователю.

## Топология графа

**Упрощение vs high-level plan**: план предлагает 4 ноды (analyze → agent ↔ tools → generate). Рекомендую **3 ноды** — без отдельного `analyze_node`. Причина: "нужен ли поиск?" — это решение, которое LLM принимает в `agent_node` через system prompt + `bind_tools()`. Отдельная нода удваивает вызовы LLM без выигрыша.

```
START → agent ──(tool_calls)──→ tools ──→ agent (loop)
                │                           │
                │                    (iteration >= 5)
                │                           │
                └──(no tool_calls)──→ generate → END
```

- **agent**: LLM с привязанными инструментами. Решает: искать / уточнить / ответить.
- **tools**: `ToolNode` выполняет инструменты, инкрементирует `iteration_count`.
- **generate**: Финальный LLM-вызов — форматирует ответ с источниками. Отдельная нода нужна, чтобы надёжно применить правило "отвечай только из базы знаний".

## 2. `agent/prompts.py`

Два промпта как строковые константы:

**SYSTEM_PROMPT** (для agent_node):

- Роль: персональный помощник по базе знаний Obsidian
- Правило: отвечать ТОЛЬКО из базы знаний, не использовать общие знания LLM
- Инструкции по `bm25_terms`: когда и как извлекать жёсткие термины (даты, аббревиатуры, имена)
- **Относительные даты**: если вопрос содержит «вчера», «на этой неделе», «в прошлом месяце» и т.п. — СНАЧАЛА вызвать `get_current_date()`, вычислить конкретную дату, затем использовать её при поиске и интерпретации результатов. Пример: «Какие задачи по даче были выполнены вчера?» → (1) `get_current_date` → 15.04.2026, (2) вчера = 14.04.2026, (3) `search_knowledge_base("задачи дача")`, (4) в ответе учесть только задачи с датой 14.04.2026
- Когда НЕ искать: прямой вопрос о дате → `get_current_date`, HUB → `create_hub_note`, small talk → ответ без инструментов
- Лимит: не более 3 поисков по одному вопросу
- Формат: markdown + "Источники: файл1.md, файл2.md"

**GENERATE_PROMPT** (для generate_node):

- Инструкция: сформировать финальный ответ на основе результатов инструментов
- Правило: только информация из KB, перечислить источники, markdown

Миграция на Langfuse — позже: `prompts.py` станет тонкой обёрткой с fallback на константы.

## 4. `agent/nodes.py`

### `agent_node(state) -> dict`

- `get_llm().bind_tools(get_tools())` → `invoke(messages)`
- System prompt вставляется один раз в `ask()` (не в ноде, чтобы не дублировать)

### `tool_node_with_counter(state) -> dict`

- Оборачивает `ToolNode(tools, handle_tool_errors=True).invoke(state)`
- Добавляет `iteration_count + 1` к результату
- `handle_tool_errors=True` — если Qdrant упал, ошибка станет `ToolMessage`, не крэш

### `generate_node(state) -> dict`

- LLM-вызов с `GENERATE_PROMPT` как system message
- Формирует финальный ответ с источниками

### `check_iteration_limit(state) -> str`

- Routing-функция: `iteration_count >= max_iterations` → `"generate"`, иначе → `"agent"`

## 5. `agent/graph.py`

### `_build_graph() -> StateGraph`

```
builder.add_node("agent", agent_node)
builder.add_node("tools", tool_node_with_counter)
builder.add_node("generate", generate_node)

START → "agent"
"agent" → tools_condition → {"tools": "tools", END: "generate"}
"tools" → check_iteration_limit → {"agent": "agent", "generate": "generate"}
"generate" → END
```

### `get_graph() -> CompiledGraph`

- Синглтон с `MemorySaver` (для short-term memory через `thread_id`)
- `recursion_limit = max_iterations * 2 + 5` — страховка от бесконечного цикла

### `ask(question, thread_id=None) -> AgentResponse`

- Главная точка входа
- Создаёт initial state: `[SystemMessage(SYSTEM_PROMPT), ("user", question)]`, `iteration_count=0`
- Вызывает `graph.invoke()` с `thread_id` в config
- Извлекает ответ из последнего AI-сообщения
- Парсит источники (`_extract_sources`) — regex по "Источники:"
- Считает чанки (`_count_chunks`) — по tool messages
- Возвращает `AgentResponse`
- При ошибке — `AgentResponse(has_answer=False, answer="Ошибка...")`

### CLI: `python -m agent.graph "вопрос"`

## Риски и mitigation

### nanogpt tool calling

Не подтверждено, что nanogpt проксирует параметр `tools` корректно. Сама модель `gpt-4.1-mini` поддерживает function calling.

- **Шаг 0 (до реализации)**: мини-тест `bind_tools()` + `invoke()` с одним tool
- **Если работает**: используем стандартный `bind_tools()`
- **Если НЕ работает**: пользователь переключится на агрегатор с поддержкой tools API. Fallback-код не нужен.

### Остальное

- **generate_hub_files.py**: хардкод пути. Рефакторинг простой, но нужно не сломать standalone-запуск скрипта.
- **MemorySaver**: данные теряются при перезапуске. Для продакшена — `SqliteSaver`.
