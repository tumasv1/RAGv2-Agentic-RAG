## ADR-0013: Интеграция MCP-сервера для работы с заметками Obsidian

**Статус:** принято

**Дата:** 27.05.26

---

## Контекст

Агент умеет отвечать на вопросы по базе знаний, но не мог читать заметки
целиком, добавлять в них информацию или создавать новые. Инструмент
`manage_note` был обозначен в ранних планах как «planned», но не реализован.

Параллельно автором проекта был разработан отдельный MCP-сервер
(`/Users/mikhail/Projects/MCP/MCP-Obsidian`) для другого агента.
Тот сервер предоставляет 8 тулзов (`list_vault`, `find_note`, `read_note`,
`search_notes`, `create_note`, `append_to_note`, `update_note`, `get_templates`)
и общается с Obsidian **через WebDAV** — внешний HTTP-сервис, который
синхронизирует vault с мобильным/десктопным клиентом Obsidian.

В RAGv2 vault уже примонтирован как `/vault:rw` в Docker Compose (том
`obsidian_vault`), а переменная `OBSIDIAN_VAULT=/vault` задана в `.env`.
Подключаться к WebDAV изнутри того же контейнера — лишний сетевой слой.

Задача: дать агенту возможность читать конкретные заметки целиком и
модифицировать их (добавить запись, переписать, создать новую).

---

## Варианты

**Вариант А: Подключиться к существующему WebDAV MCP-серверу**

Запустить `MCP-Obsidian` как внешний процесс (или переиспользовать в том же
контейнере), передав ему WebDAV-кредентиалы.
- Плюсы: готовый код, ничего писать не нужно
- Минусы: лишний сетевой слой через WebDAV внутри одного хоста; нужны
  кредентиалы в env; сервер написан под другую инфраструктуру (другой агент,
  другой Docker Compose); зависимость от WebDAV-сервиса — если он упал,
  агент не может работать с заметками

**Вариант Б: Адаптировать MCP-сервер для работы напрямую с ФС (принято)**

Создать пакет `mcp_obsidian/` внутри RAGv2, взяв за основу стиль, имена и
docstring'и тулзов оригинального `MCP-Obsidian/server.py`, но заменив
транспорт WebDAV → локальная ФС через `VaultFs`.

- Плюсы: прямой доступ к смонтированному тому без сетевого слоя; vault-root
  берётся из уже существующего `cfg.obsidian_vault` — новых env-переменных
  нет; независимость от WebDAV-сервиса; docstring'и тулзов полностью
  переиспользуются — LLM получает те же tool-spec подсказки
- Минусы: дублирование кода относительно оригинального сервера; при обновлении
  оригинала нужно синхронизировать вручную

**Вариант В: Написать собственный MCP-сервер с нуля**

- Плюсы: нет зависимости от дизайна оригинала
- Минусы: больше работы; оригинальный сервер уже отлажен — нет смысла
  изобретать велосипед

---

## Решение

Выбираем **Вариант Б**: адаптация оригинального `MCP-Obsidian` для локальной ФС.

Реализация:

- **`mcp_obsidian/fs_client.py`** — `VaultFs` с path-traversal защитой
  (`_normalize`): все пути проверяются на выход за пределы vault-root.
  Скрытые файлы (`.obsidian`, `.trash`, `.DS_Store`) фильтруются в листингах.
- **`mcp_obsidian/server.py`** — FastMCP stdio-сервер. Тулзы (7 из 8
  оригинальных):
  - `list_vault`, `find_note`, `read_note`, `create_note`, `append_to_note`,
    `update_note`, `get_templates`
  - **`search_notes` намеренно не реализован**: в RAGv2 есть более мощный
    гибридный поиск с реранкером (`search_knowledge_base`)
- **`agent/mcp_tools.py`** — lazy async loader на `AsyncExitStack`: удерживает
  MCP-сессию (subprocess) живой в event loop uvicorn, избегая кросс-луп
  дедлоков.

Сервер запускается как stdio child process внутри того же контейнера:
```
command: sys.executable
args: ["-m", "mcp_obsidian.server"]
transport: stdio
```

## Решение 2: API create_note — template_name вместо frontmatter_data

### Проблема

Первая версия `create_note` принимала `frontmatter_data: dict` — агент должен
был передать полный словарь frontmatter (поля `type`, `created`, `tags` и т.д.).
На практике модель **всегда передавала `{}`** (пустой dict), хотя в текстовом
ответе правильно описывала нужный frontmatter. Инструмент создавал файлы
без frontmatter.

Диагностика: LLM плохо справляется с конструированием dict-параметров в
tool call — намного легче передать строки.

### Решение

Заменить `frontmatter_data: dict` на два строковых параметра:
- `template_name: str` — точное имя шаблона из `get_templates()`
- `note_type: str` — одно значение из списка типов шаблона

Сервер сам читает шаблон, подставляет `created: DD.MM.YY` из системных
часов и строит YAML frontmatter. Агент не конструирует dict.

Результат: с первого вызова файл создаётся с правильным frontmatter.

## Решение 3: Выбор папки при создании заметки

### Проблема

Агент создавал заметки в корне vault (`/vault/note.md`) вместо подпапок.
`list_vault` возвращал только один уровень — агент не видел структуру
без нескольких последовательных вызовов.

### Решение

1. Добавить параметр `depth: int = 1` в `list_vault` — при `depth=2` каждый
   dir-entry содержит поле `children[]` с вложенными папками. Один вызов
   `list_vault("Основное", depth=2)` даёт полную двухуровневую структуру.

2. В `SYSTEM_PROMPT` — явные правила:
   - никогда не создавать в корне vault или прямо в `Основное/`
   - обязательно вызывать `list_vault("Основное", depth=2)` перед выбором папки
   - если назначение очевидно → выбрать самостоятельно и сообщить пользователю
   - при малейшем сомнении → задать вопрос с пронумерованными вариантами
     и дождаться ответа перед вызовом `create_note`

## Решение 4: Async-только MCP-тулзы и переход на AsyncSqliteSaver

### Проблема

`langchain-mcp-adapters` создаёт `StructuredTool` с только `_arun` (async),
без `_run` (sync). `ToolNode.invoke()` вызывает `_run` → ошибка.

Одновременно: при переходе к async tool calls понадобился async checkpointer.
`SqliteSaver` не поддерживает `graph.ainvoke()`.

### Решение

- Граф полностью переведён на async: `graph.ainvoke()`, ноды — `async def`
- `tool_node_with_counter` — `async def`, вызывает `await _get_tool_node().ainvoke(state)`
- `SqliteSaver` → `AsyncSqliteSaver` + `aiosqlite`
- `_ensure_graph()` — async синглтон с `asyncio.Lock`

Инициализация `AsyncSqliteSaver`:
```python
conn = await aiosqlite.connect(str(db_path))
_checkpointer = AsyncSqliteSaver(conn=conn)
```
⚠️ `AsyncSqliteSaver.from_conn_string()` — это `@asynccontextmanager`, не coroutine.

## Последствия

- Агент умеет: читать заметку целиком, добавлять в раздел, переписывать,
  создавать новую по шаблону с правильным frontmatter
- `search_notes` не реализован — поиск содержимого идёт через
  `search_knowledge_base` (гибридный поиск с реранкером)
- WebDAV-сервис в Docker Compose остаётся для мобильного/десктопного
  клиента Obsidian; из RAGv2 в него больше не ходим
- HITL (подтверждение пользователя перед записью) — **отложено**; агент
  спрашивает про папку, но не запрашивает подтверждения перед записью файла
- При обновлении оригинального `MCP-Obsidian` — проверить, не нужно ли
  синхронизировать `mcp_obsidian/server.py`
- Автореиндексация vault после `create_note`/`update_note` — не реализована;
  пользователь триггерит `/api/reindex` вручную
