## ADR-0003: Раздельный BM25-запрос и опциональный реранкер

**Статус:** принято

**Дата:** 14.04.26

---

## Контекст

После реализации гибридного поиска (Фаза 1) выявились две проблемы:

1. **BM25 шумит на natural language запросах.** Когда один и тот же текст ("Какие задачи на эту неделю?") идёт и в dense (E5), и в sparse (BM25), BM25-ветка возвращает нерелевантные чанки по совпадению общих слов ("на", "эту"). RRF усредняет ранги, и шум из BM25 портит финальный список.

2. **Нет этапа precision-фильтрации.** Bi-encoder (E5) оценивает запрос и документ по отдельности. Cross-encoder видит их вместе и точнее судит о релевантности, но в pipeline его не было.

---

## Варианты

### BM25-запрос

**Вариант А:** Один запрос для обеих веток (как было)
- Плюсы: простота, один параметр
- Минусы: BM25 шумит на обычных вопросах, RRF-скоры размываются

**Вариант Б:** Раздельные запросы — LLM извлекает «жёсткие термины» для BM25
- Плюсы: BM25 работает по назначению (exact match дат, аббревиатур, кодов); dense-only для обычных вопросов чище
- Минусы: зависимость от качества LLM-экстракции; дополнительный параметр в API

### Реранкер

**Вариант А:** sentence-transformers CrossEncoder (PyTorch)
- Плюсы: проверенный паттерн из RAG v1, простой API
- Минусы: медленнее на CPU (PyTorch), `bge-reranker-base` был слишком медленным в другом проекте

**Вариант Б:** fastembed TextCrossEncoder (ONNX)
- Плюсы: ONNX-оптимизация быстрее на CPU, fastembed уже установлен
- Минусы: ограниченный набор моделей, менее документированный API

### Пороги score

**Вариант А:** Один `score_threshold` для всех этапов
- Плюсы: простой конфиг
- Минусы: шкалы разные (cosine 0–1, BM25 ~1–5, cross-encoder ~0–1), один порог не подходит всем

**Вариант Б:** Раздельные пороги для каждого этапа
- Плюсы: точная настройка под каждую шкалу, не перегружаем LLM слабыми чанками
- Минусы: больше параметров в конфиге

---

## Решение

### 1. Раздельный BM25-запрос

Выбираем **Вариант Б**. Сигнатура инструмента: `search(query, bm25_terms=None)`.

- `bm25_terms=None` → dense-only поиск (пропускаем BM25)
- `bm25_terms="ШР 30/2024"` → гибридный поиск (dense + BM25 по терминам + RRF)

LLM-агент на первой ноде анализирует запрос пользователя и извлекает жёсткие термины (даты, аббревиатуры, коды, имена собственные). Пустая строка трактуется как None.

### 2. Реранкер — fastembed ONNX

Выбираем **Вариант Б**. Модель: `jinaai/jina-reranker-v2-base-multilingual` (~1.1 ГБ ONNX).

- Мультиязычный (100+ языков, русский)
- ONNX быстрее PyTorch на CPU
- fastembed уже в зависимостях (через langchain-qdrant)
- Pure re-sort: score реранкера полностью заменяет score retrieval (не взвешенная комбинация)
- Ленивый синглтон: модель грузится только при `use_reranking: true`

### 3. Раздельные пороги

Выбираем **Вариант Б**. Три порога в `SearchConfig`:

- `dense_score_threshold` — cosine similarity (0–1), фильтрация в Qdrant Prefetch
- `sparse_score_threshold` — BM25 score, фильтрация в Qdrant Prefetch
- `reranker_score_threshold` — cross-encoder score, фильтрация после реранкинга

Дефолты = 0.0 (не фильтруем). Подбираются по RAGAS-бенчмаркам.

---

## Последствия

- Dense-only поиск стал чище — нет BM25-шума на обычных вопросах
- BM25 стал precision-инструментом для точных совпадений (даты, коды)
- Реранкер даёт второй уровень оценки релевантности (cross-encoder видит query+doc вместе)
- Раздельные пороги позволяют не перегружать LLM слабыми чанками
- Нужен промпт для LLM-агента: инструкция по извлечению жёстких терминов
- При `use_reranking: true` первый запрос медленнее (~10 сек на загрузку модели)
- RAM с реранкером: ~2.2–2.7 ГБ (вместо ~1.65 ГБ без)
