## ADR-0002: Инфраструктурные и технологические решения

**Статус:** принято

**Дата:** 08.04.26

---

## Контекст

После принятия ADR-0001 (общая архитектура агента) остался ряд технологических решений, без которых невозможно приступить к реализации: выбор векторной БД, стратегия гибридного поиска, веб-фреймворк, управление конфигурацией и стратегия инкрементальной индексации.

Также были выявлены противоречия между требованиями и ADR-0001, которые необходимо зафиксировать как решённые.

---

## Решения

### 1. Векторная база данных: Qdrant

**Варианты:**

**Вариант А: ChromaDB + rank_bm25**
- Плюсы: уже использовался в RAG v1, простой API
- Минусы: BM25 реализуется отдельной библиотекой, нет нативного гибридного поиска, два разных индекса нужно поддерживать и синхронизировать

**Вариант Б: Qdrant** (выбран)
- Плюсы: нативная поддержка sparse vectors (BM25) и dense vectors в одной коллекции, встроенный гибридный поиск с Reciprocal Rank Fusion, production-ready
- Минусы: новый инструмент для освоения, чуть сложнее в первоначальной настройке

**Решение:** Qdrant. Нативная поддержка гибридного поиска избавляет от необходимости синхронизировать два индекса и вручную объединять результаты.

---

### 2. Гибридный поиск: Reciprocal Rank Fusion (RRF)

Стратегия объединения результатов dense (embedding) и sparse (BM25) поиска — **Reciprocal Rank Fusion**. Qdrant поддерживает RRF из коробки через параметр `fusion` в запросе.

RRF объединяет ранжированные списки результатов от разных поисковых методов, отдавая приоритет документам, которые занимают высокие позиции в нескольких списках одновременно. Не требует нормализации scores, что делает его устойчивым к различиям в масштабах оценок между dense и sparse поиском.

---

### 3. Веб-фреймворк: FastAPI

**Варианты:**

**Вариант А: Streamlit / Gradio**
- Плюсы: быстрый прототип, минимум кода для UI
- Минусы: ограниченный контроль над UI, сложно сделать debug dashboard, не подходит для production API

**Вариант Б: FastAPI** (выбран)
- Плюсы: полный контроль над API, async из коробки, автоматическая документация OpenAPI, хорошая экосистема, возможность изучить фреймворк
- Минусы: фронтенд нужно делать отдельно (шаблоны Jinja2 или отдельный SPA)

**Решение:** FastAPI. Помимо технических плюсов — цель изучить фреймворк на практике.

---

### 4. Инкрементальная индексация: по дате изменения файла

**Стратегия определения изменённых файлов:** сравнение `mtime` (дата последнего изменения) файла с датой предыдущей индексации.

Этот подход уже был опробован и работал стабильно в проекте RAG v1 (`/Users/mikhail/RAG/ingestbatch.py`). Дата последней индексации хранится в файле состояния (или в логе индексации).

---

### 5. Переменные окружения (.env)

Зафиксирован начальный набор переменных:

| Переменная | Назначение |
|---|---|
| `NANO_GPT_API_KEY` | API-ключ nanogpt |
| `NANO_GPT_BASE_URL` | Базовый URL nanogpt API |
| `NANO_GPT_MODEL` | Модель LLM по умолчанию |
| `OBSIDIAN_VAULT` | Путь к хранилищу Obsidian |
| `TELEGRAM_BOT_TOKEN` | Токен Telegram бота |

Список будет дополняться по мере реализации (Qdrant URL, Langfuse ключи и т.д.).

---

### 6. Конфигурация: config.yaml

Необходимо определить schema для `config.yaml`. Минимальный набор параметров (из требований §5):

- Модель LLM и температура
- Максимальное количество чанков в контексте LLM
- Пороговый score для фильтрации чанков
- Максимальное число итераций агента
- Путь к базе знаний Obsidian (дублирует .env, но может переопределяться)

Schema будет зафиксирована в отдельном ADR или в виде Pydantic-модели при реализации.

## Последствия

- Qdrant нужно будет развернуть локально (Docker или embedded mode) — учесть при планировании Фазы 0
- FastAPI потребует отдельного решения по фронтенду (Jinja2 шаблоны для MVP vs отдельный SPA позже)
- `.env` добавлен в `.gitignore` для защиты секретов
- Инкрементальная индексация по mtime — простой и проверенный подход, но не ловит переименования файлов (приемлемый trade-off)
