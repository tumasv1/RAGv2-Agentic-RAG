## ADR-0006: Архитектура модуля оценки качества (eval/)

**Статус:** принято

**Дата:** 17.04.26

---

## Контекст

После завершения retriever/ (Фаза 1) и agent/ (Фаза 2) нужен инструмент для
объективной оценки качества RAG-пайплайна и подбора параметров (пороги score,
стратегия чанкинга, реранкер). Без цифр невозможно принять решение, например:
«`chunk_size=800` лучше или хуже `chunk_size=1700`?»

Требования из спецификации: RAGAS-метрики (faithfulness, precision, recall,
relevancy), golden set, автоматическое сравнение стратегий чанкинга.

---

## Решение 1: eval/ тестирует retriever + LLM напрямую, минуя агент

### Варианты

**Вариант А:** eval/ вызывает `agent.graph.ask()` — тестирует полный пайплайн
- Плюсы: тест ближе к реальному пользовательскому сценарию
- Минусы: любые изменения в логике агента (промпты, routing) влияют на метрики;
  нельзя изолировать retrieval-качество от оркестрации; медленнее (5 итераций агента)

**Вариант Б (принято):** eval/ вызывает `retriever.search.search()` + LLM напрямую
- Простой пайплайн: `search(question)` → `LLM(question, contexts)` → RAGAS-оценка
- Плюсы: изолирует retrieval-качество; независимость от agent/; легко подменить
  search function (нужно для compare_splitters); быстрее
- Минусы: не тестирует логику агента (multi-hop, bm25_terms, итерации)

### Решение

Выбираем **Вариант Б**. Цель eval/ — измерить качество retrieval и генерации
по контексту, а не качество агентской оркестрации. Для тестирования агента
есть notebooks/debug.ipynb и интеграционные тесты.

## Решение 2: 5 файлов вместо 2 (runner + metrics + report + 2 CLI)

### Контекст

High-level plan описывал 2 файла: `eval_ragas.py` и `compare_splitters.py`.
В RAG v1 всё было в одном файле (~400 строк) — это работало, но при добавлении
compare_splitters код прогона и отчётов пришлось бы дублировать.

### Варианты

**Вариант А (по плану):** 2 монолитных файла
```
eval_ragas.py          # load + build_dataset + compute + write_report
compare_splitters.py   # та же логика + управление коллекциями
```
- Плюсы: меньше файлов, проще для начинающего
- Минусы: дублирование build_dataset, compute_metrics, write_report в обоих файлах;
  при изменении формата отчёта нужно менять в двух местах

**Вариант Б (принято):** 3 переиспользуемых модуля + 2 CLI
```
runner.py              # load_golden_set, generate_answer, run_golden_set, EvalDataset
metrics.py             # compute_metrics (RAGAS), setup_ragas_llm/embeddings
report.py              # write_report (6 модульных секций, светофоры)
eval_ragas.py          # CLI: runner → metrics → report
compare_splitters.py   # CLI: индексация → runner → metrics → сводный отчёт
```
- Плюсы: compare_splitters переиспользует runner и metrics без дублирования;
  формат отчёта меняется в одном месте (report.py)
- Минусы: больше файлов (5 вместо 2)

### Решение

Выбираем **Вариант Б**. Дублирование кода дороже, чем лишние файлы.

## Решение 3: search_fn как инжектируемая зависимость в run_golden_set()

### Контекст

compare_splitters создаёт временные Qdrant-коллекции для каждой стратегии и
должен искать по ним, а не по основной коллекции. Нужно передать альтернативную
функцию поиска в run_golden_set().

### Варианты

**Вариант А:** добавить параметр `collection_name: str` в `retriever.search.search()`
- Плюсы: чисто — одна функция для всех коллекций
- Минусы: меняем retriever/search.py ради нужды eval/; search() заточен под синглтон

**Вариант Б (принято):** параметр `search_fn: Callable` в run_golden_set()
```python
def run_golden_set(
    cases: list[dict],
    search_fn: Callable[[str], list[SearchResult]] | None = None,
) -> EvalDataset:
    if search_fn is None:
        search_fn = retriever.search.search  # дефолт
```
- Плюсы: retriever/search.py не меняем; compare_splitters создаёт свою функцию;
  паттерн dependency injection — легко тестировать с mock
- Минусы: compare_splitters дублирует часть логики search() для временной коллекции

**Вариант В:** временно заменять синглтон `_qdrant_store` в indexer.py
- Плюсы: поиск по временной коллекции через тот же `search()`
- Минусы: глобальное состояние; опасно при многопоточности; неочевидный side effect

### Решение

Выбираем **Вариант Б**. Dependency injection — стандартный паттерн для
изолированного тестирования. Не трогаем retriever/.

## Решение 4: mhts_only — chunk_size=100_000 вместо отдельной функции

### Контекст

Стратегия `mhts_only` — только MarkdownHeaderTextSplitter без дорезки RCTS.
Нужно реализовать это, не изменяя chunker.py (принцип: eval/ не трогает retriever/).

### Варианты

**Вариант А:** добавить флаг `use_rcts: bool` в `chunk_file()` или `IngestConfig`
- Плюсы: явный флаг, семантически понятный
- Минусы: изменение chunker.py ради нужды eval/; разрастание интерфейса

**Вариант Б (принято):** chunk_size=100_000 (RCTS не разрезает, т.к. секции меньше)
- Идея: RCTS с лимитом 100 000 символов никогда не разрежет реальные секции
  (средний чанк ~500 символов), то есть RCTS де-факто становится no-op
- Плюсы: chunker.py не меняем; 1 строка кода; без флагов
- Минусы: неочевидный "трюк" — нужен комментарий

**Вариант В:** написать `chunk_file_mhts_only()` в compare_splitters.py
- Плюсы: явная реализация
- Минусы: дублирование большей части кода chunker.py

### Решение

Выбираем **Вариант Б**. Трюк прост и легко объясняется в комментарии к стратегии.
`mhts_only: chunk_size=100_000` — читается как "отключить RCTS".

## Последствия

- eval/ полностью независим от agent/ — можно запускать после Фазы 1
- run_golden_set() — точка входа для любого eval-сценария (можно добавлять новые стратегии без изменения runner.py)
- Отчёты в `reports/` пишутся в append-режиме — каждый запуск добавляет секцию с timestamp
- compare_splitters удаляет временные коллекции после оценки (qdrant_data/ не разрастается)
- **Следующий шаг:** запустить полное сравнение 4 стратегий на всех 18 кейсах: `python -m eval.compare_splitters`
