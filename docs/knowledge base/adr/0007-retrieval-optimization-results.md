## ADR-0007: Оптимизация retrieval — выбор стратегии чанкинга, параметров поиска и LLM-провайдера

**Статус:** принято

**Дата:** 20.04.26

---

## Контекст

После реализации eval-модуля (ADR-0006) проведена серия экспериментов для подбора
оптимальных параметров RAG-пайплайна. Тестировались: стратегии чанкинга, пороги
и лимиты dense-поиска, гибридный поиск (dense + BM25), реранкер, LLM-провайдер
и модель.

Инструменты:
- `python -m eval.compare_splitters --persist --index-only` — построить постоянные коллекции
- `python -m eval.eval_ragas --strategy <name>` — eval по конкретной коллекции
- `eval/golden_set.yaml` — 18 кейсов (основной golden set)
- `eval/golden_set_bm25.yaml` — кейсы с датами и аббревиатурами (для BM25)

---

## Решение 1: Стратегия чанкинга → large (MHTS + RCTS 2500/300)

### Протестированные стратегии

| Стратегия | chunk_size | overlap | use_mhts | Описание |
|-----------|-----------|---------|----------|---------|
| baseline  | 1700 | 200 | True  | Текущая |
| mhts_only | 100000 | 0 | True  | Только MHTS |
| small     | 800  | 100 | True  | Мелкие чанки |
| large     | 2500 | 300 | True  | Крупные чанки |
| rcts_only | 1700 | 300 | False | Только RCTS |

### Победитель: `large`

Стратегия `large` (MHTS + RCTS, chunk_size=2500, overlap=300) показала наилучшие
результаты по совокупности RAGAS-метрик и оценок LLM-судьи.

**Почему крупные чанки лучше:**
- Context recall растёт: крупный чанк чаще содержит полный ответ
- Для личной базы знаний (~500 символов средний чанк) 2500 = ~5 абзацев — оптимально
- Overlap=300 снижает потери на границах секций

### Решение

Переключить prod-коллекцию на стратегию `large`. Переиндексировать vault.

## Решение 2: Параметры dense-поиска → max_chunks=15, dense_score_threshold=0.79

### Варианты

**Baseline:** max_chunks=10, fetch_k=10, dense_score_threshold=0.75

**Принято:** max_chunks=15, fetch_k=15, dense_score_threshold=0.79

### Сравнение метрик

| Метрика           | Baseline | Принято      |
| ----------------- | -------- | ------------ |
| Faithfulness      | 0.805    | 0.758        |
| Answer Relevancy  | 0.738    | **0.792**    |
| Context Precision | 0.719    | **0.733**    |
| Context Recall    | 0.676    | **0.723**    |
| LLM-судья (0-3)   | 1.61 / 3 | **1.81 / 3** |

**Baseline pipeline (max_chunks=10, fetch_k=10, dense_score_threshold=0.75):**

| Параметр | Значение |
|----------|----------|
| max_chunks | 10 |
| fetch_k | 10 |
| dense_score_threshold | 0.75 |
| sparse_score_threshold | 0.0 |
| reranker_score_threshold | -3.3 |
| use_reranking | False |

**Финальный pipeline (max_chunks=15, fetch_k=15, dense_score_threshold=0.79):**

| Параметр                 | Значение |
| ------------------------ | -------- |
| max_chunks               | 15       |
| fetch_k                  | 15       |
| dense_score_threshold    | 0.79     |
| sparse_score_threshold   | 1.35     |
| reranker_score_threshold | -3.3     |
| use_reranking            | False    |

### Интерпретация

Faithfulness немного упал (0.805 → 0.758): больше чанков → чуть больше шума в
контексте, LLM иногда добавляет что-то лишнее. Но остальные три метрики и
LLM-судья выросли заметно. Судья (+0.20) важнее как интегральная оценка.

Порог 0.79 отсекает слабые совпадения, оставляя только уверенные. При 0.75 в
контекст попадал «мусор», который снижал точность ответа.

### Решение

Принимаем max_chunks=15, fetch_k=15, dense_score_threshold=0.79.

## Решение 3: BM25 — bm25_top_k=3, sparse_score_threshold=1.35

### Контекст

BM25 предназначен для точного поиска по датам, аббревиатурам, кодам (ШР 30/2024,
2024-03-15 и т.п.). Создан отдельный `eval/golden_set_bm25.yaml` с кейсами,
где нужны точные совпадения по терминам.

### Параметры

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| bm25_top_k | 3 | BM25 — точечный инструмент, 3 кандидатов достаточно |
| sparse_score_threshold | 1.35 | Отсекает слабые BM25-совпадения |

**Почему bm25_top_k=3, а не больше:**
- BM25 работает с жёсткими терминами, которые LLM-агент явно извлекает из вопроса
- Больше BM25-кандидатов = больше шума при RRF-слиянии с dense
- 3 кандидата от BM25 + до 15 от dense → RRF даёт нужную смесь

### Решение

bm25_top_k=3 и sparse_score_threshold=1.35 зафиксированы в config.yaml.
BM25 активируется агентом только при наличии жёстких терминов в вопросе.

## Решение 4: Реранкер → отключить (use_reranking=False)

### Контекст

Тестировался кросс-энкодер `jinaai/jina-reranker-v2-base-multilingual`.
Функционально работает корректно.

### Проблема

Реранкер на CPU значительно увеличивает время ответа. При текущем уровне качества
dense-поиска (судья 1.81/3) дополнительные метрики по реранкеру не анализировались —
нет смысла платить временем, если retrieval уже достаточно точен.

### Решение

`use_reranking: false`. Вернуться к реранкеру, если в будущем появятся кейсы,
где dense+BM25 явно ошибается, а скорость ответа менее критична.

## Решение 5: LLM-провайдер → OpenRouter вместо nanogpt

### Варианты

**nanogpt:** использовался с момента старта проекта (см. ADR-0002)
- Минус: скорость ответа в ~2 раза ниже на тех же вопросах

**OpenRouter (принято):**
- Плюс: скорость ответа в ~2 раза выше при той же модели
- Плюс: широкий выбор моделей (будущая гибкость)
- Модель: `openai/gpt-4.1-mini`

### Выбор модели

Тестировались `gpt-4.1-mini` и `gpt-4o-mini`:
- `gpt-4o-mini` в некоторых кейсах пропускает полезную информацию из контекста
- `gpt-4.1-mini` стабильно использует весь контекст → остаётся основной моделью

### Решение

Переключить провайдер на OpenRouter. Модель: `openai/gpt-4.1-mini`.
Переменные в `.env`: `NANO_GPT_BASE_URL`, `NANO_GPT_API_KEY` → обновлены на OpenRouter.

## Итоговые параметры пайплайна

```yaml
# config.yaml

search:
  max_chunks: 15
  fetch_k: 15
  bm25_top_k: 3
  dense_score_threshold: 0.79
  sparse_score_threshold: 1.35
  reranker_score_threshold: -3.3
  use_reranking: false

ingest:
  chunk_size: 2500
  chunk_overlap: 300
  use_mhts: true
```

## Последствия

- Prod-коллекцию `obsidian_notes` нужно переиндексировать с параметрами стратегии `large`
- `eval/golden_set_bm25.yaml` остаётся отдельным набором для проверки BM25-кейсов
- Реранкер можно вернуть позднее без изменения кода — достаточно `use_reranking: true`
- **Следующий шаг:** переиндексировать vault (`python -m retriever.indexer`)
