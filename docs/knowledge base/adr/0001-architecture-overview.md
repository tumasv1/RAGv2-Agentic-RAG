## ADR-0001: Общая архитектура агента RAG v2

**Статус:** принято

**Дата:** 07.04.26

---

## Контекст

Нужно построить персональный ИИ-помощник, который отвечает на вопросы на основе личной базы знаний Obsidian. Помощник должен работать без интернета (кроме вызовов LLM), на слабом домашнем сервере (AMD Ryzen 5 6600H, 8 ГБ RAM, без GPU).

Уже существующий код в проекте:
- `golden_set.yaml` — Набор данных для измерения качества с вопросами и целевыми ответами.
- `check_frontmatter.py` — проверка наличия frontmatter во всех заметках (рабочий скрипт)
- `generate_golden_set.py` — генерация синтетических тест-кейсов через LLM (зависит от `rag_pipeline.py` из `/Users/mikhail/RAG/`)

Базовый RAG уже работал (`/Users/mikhail/RAG/rag_pipeline.py`). RAG v2 — переписывается с нуля с агентной архитектурой.

---

## Варианты

### Агентный фреймворк

**Вариант А: LangGraph** (выбран)
- Плюсы: граф состояний, встроенный HITL, subgraphs для модульности, retry/fallback из коробки
- Минусы: сложнее для новичка, чем чистый Python

**Вариант Б: Чистый Python**
- Плюсы: полный контроль, просто
- Минусы: надо руками реализовать граф, прерывания, retry — слишком много бойлерплейта

### LLM-доступ

**Вариант А: nanogpt aggregator** (выбран)
- Плюсы: уже используется в rag_pipeline.py, OpenAI-совместимый API, экономия за счёт агрегации
- Минусы: внешняя зависимость

**Вариант Б: Ollama (локально)**
- Плюсы: полностью офлайн
- Минусы: 8 ГБ RAM не хватит для достаточно сильной модели

### Конфигурация

**Вариант А: config.yaml + .env** (выбран)
- Плюсы: просто, прозрачно, стандартно
- Минусы: нужен рестарт при изменениях

**Вариант Б: config.yaml + веб-UI для редактирования**
- Плюсы: можно менять без рестарта
- Минусы: избыточно для старта

## Решение

Строим агента на **LangGraph** по паттерну ReAct (рассуждай → действуй → наблюдай).

### Ноды vs. Инструменты — важное разграничение

**Нода** — это шаг в графе рассуждений (что агент делает на каждом этапе).
**Инструмент** — это функция, которую агент вызывает во время работы.

Все инструменты вызываются через одну ноду `tool_node`. LLM внутри ноды `agent` сам решает, какой инструмент нужен в данный момент — отдельная нода на каждый инструмент не нужна.

### Граф агента

```
START
  ↓
[analyze]       нужен ли вообще поиск по базе знаний?
  ↓ да          ↓ нет (общий вопрос / дата / управление заметками)
  ↓             ↓
  └─────────→ [agent]   LLM смотрит на историю и выбирает инструмент
                  ↓ вызов инструмента
             [tool_node]  выполняет один из 6 инструментов:
                  │  • search_knowledge_base(query)
                  │  • get_current_date()
                  │  • create_hub_note()
                  │  • get_doc_metadata(doc)
                  │  • manage_note(action, note)  ← HITL для write/delete
                  │  • check_frontmatter()
                  ↓ результат
             [agent]   достаточно данных для ответа?
                  ↓ нет — снова вызывает инструмент (до 5 итераций)
                  ↓ да
             [generate]   финальный ответ с указанием источников
                  ↓
                END
```

### HITL (Human-in-the-Loop)

Перед выполнением `manage_note` с action=`write` или `delete` граф прерывается (`interrupt`). Пользователь подтверждает действие — только после этого выполнение продолжается.

### Инструменты агента

| Инструмент | Описание | Приоритет |
|---|---|---|
| `search_knowledge_base(query)` | Гибридный поиск по базе знаний | Обязательный |
| `get_current_date()` | Текущая дата и время | Обязательный |
| `create_hub_note()` | Создание/обновление навигационных hub-заметок | Обязательный |
| `get_doc_metadata(doc)` | Метаданные чанка по имени файла | В последнюю очередь |
| `manage_note(action, note)` | Чтение, создание, редактирование, удаление заметки | В последнюю очередь |
| `check_frontmatter()` | Проверяет наличие frontmatter у всех заметок | В последнюю очередь |

**LLM:** nanogpt aggregator (OpenAI-совместимый клиент из rag_pipeline.py)

**Embeddings:** `BAAI/bge-m3` или `intfloat/multilingual-e5-large` локально на CPU. Необходимо позже определиться какой именно выбрать. Учесть что поиск будет гибридными вместе с BM25.

**Поиск:** гибридный (embedding + BM25), reranker опционально (выбрать по результатам RAGAS)

**Конфигурация:** `config.yaml` + `.env`

**Guardrails:**
- Максимум 5 итераций на запрос
- HITL обязателен для `manage_note` с action=`write` или `delete`

**Память:** short-term (история сессии) + суммаризация через `summarize_messages()` при длинном контексте

## Последствия

- Сначала реализуем базовый граф (analyze → agent → tool_node → generate) без HITL, потом добавляем guardrails и память
- Выбор reranker'а откладывается до получения первых RAGAS-метрик
- Стратегия чанкинга выбирается через автоматическое сравнение по RAGAS (отдельный скрипт)
- Langfuse для промптов и логов подключается после того, как базовый граф заработает
