## ADR-0004: Контекстный префикс в page_content чанков

**Статус:** принято

**Дата:** 14.04.26

---

## Контекст

После индексации 2651 чанка из 400 заметок обнаружилась проблема: `page_content` содержит только текст чанка. Имя файла, иерархия заголовков, тип и теги хранятся в `ChunkMetadata`, но **не участвуют в поиске** — ни в dense-эмбеддинге (E5), ни в BM25.

Конкретный кейс: запрос "Сколько детей у Галаевой" не находил чанки из `Галаева Елена.md`, потому что фамилия упоминалась только в имени файла, а не в тексте заметки.

В RAG v1 (`ingestbatch.py:216`) имя файла препендилось к тексту: `f"Название файла: {file_name}\n{chunk_text}"` — и это работало. RAGv2 хранит метаданные богаче (heading_hierarchy, type, tags), но не использует их при индексации.

---

## Варианты

**Вариант А:** Препендить только имя файла (как в RAG v1)
- Плюсы: минимальное изменение, проверенный паттерн
- Минусы: не задействует heading_hierarchy, type, tags — теряем потенциал

**Вариант Б:** Препендить структурированный префикс (файл, путь, тип, теги)
- Плюсы: все метаданные участвуют в поиске; heading_hierarchy даёт контекст ("Контакты > Семья"); type и tags помогают семантическому матчу
- Минусы: больше токенов на чанк (~50–150 символов); структурные метки ("Файл:", "Путь:") попадают в BM25 (но с низким IDF — не шумят)

**Вариант В:** Отдельное поле для обогащённого текста (indexed_content vs page_content)
- Плюсы: оригинальный текст остаётся чистым
- Минусы: усложняет Qdrant payload и LangChain-интеграцию; два поля вместо одного

---

## Решение

Выбираем **Вариант Б** — структурированный контекстный префикс.

### Формат префикса

Включаем только непустые поля:
```
Файл: Галаева Елена
Путь: Контакты > Семья > Дети
Тип: employee
Теги: команда, HR
---
[оригинальный текст чанка]
```

### Что включаем, что нет

| Поле | В префиксе | Почему |
|------|-----------|--------|
| file_name | да | основная проблема — поиск по имени файла |
| heading_hierarchy | да | семантический контекст раздела |
| type | да | помогает dense-поиску по типу документа |
| tags | да | помогает dense-поиску по тематике |
| created | нет | дата не помогает поиску, добавляет шум в эмбеддинг; будет в форматтере контекста для LLM |

### Реализация

1. Функция `_build_context_prefix(meta)` в `retriever/chunker.py`
2. Обогащение в `chunk_file()` **после** RCTS-сплита — чтобы не съедать бюджет `chunk_size`
3. Флаг `enrich_content: bool = True` в `IngestConfig` — для A/B тестирования через RAGAS

### Payload-индексы

Добавлены `create_payload_index()` для `metadata.type`, `metadata.file_name`, `metadata.tags` в `retriever/indexer.py`. В embedded-режиме Qdrant индексы не работают (предупреждение: "Payload indexes have no effect in the local Qdrant"), но готовы для перехода на server Qdrant.

---

## Последствия

- Запрос "Галаевой" теперь находит файл `Галаева Елена.md` (dense #1, score 0.7993)
- Семантический поиск не деградировал (score 0.85+ для релевантных результатов)
- Требуется **полная переиндексация** при изменении формата префикса (2576 чанков, ~9 мин на CPU)
- Каждый чанк стал на ~50–150 символов длиннее (при chunk_size=1700 это 3–9%)
- Heading hierarchy видна в результатах поиска (например "47 неделя (17.11.25) → Планирование по Дням → Среда")
- Выявлена проблема diversity: один файл может занять 5/10 результатов → нужен per-source лимит (Batch 2)
