## ADR-0010: Parent-Child чанкинг

**Статус:** принято

**Дата:** 24.04.26

---

## Контекст

После Batch 1 (ADR-0004) чанки стали находиться лучше (имя файла в эмбеддинге), но появилась другая проблема: чанки слишком мелкие (~800 символов). Одна смысловая единица из документа разбивается на несколько чанков — и при поиске нередко находится только часть контекста, а не весь.

Конкретная ситуация: документ о сотруднике содержит связанную информацию (должность, функционал, контакты, примечания). RCTS-чанк 800 символов захватывает только один фрагмент. LLM получает неполный контекст и отвечает частично.

При увеличении `chunk_size` точность поиска падает: большой чанк «размывает» эмбеддинг, score снижается ниже порога `dense_score_threshold`.

---

## Варианты

**Вариант А:** Убрать H3 из MHTS — секции станут крупнее
- Плюсы: одна строка в конфиге
- Минусы: не решает проблему системно; зависит от структуры заметок

**Вариант Б:** Увеличить chunk_size + уменьшить dense_score_threshold
- Плюсы: простота
- Минусы: теряем точность; надо перебирать пороги заново; помогает не всем документам

**Вариант В:** Parent-Child — искать по малым children, отдавать LLM крупные parents
- Плюсы: поиск точный (малые чанки), контекст полный (крупные чанки); решает проблему системно
- Минусы: сложнее архитектура; больший индекс (дублирование текста parents); нужна полная переиндексация

---

## Решение

Выбираем **Вариант В** — Parent-Child архитектуру.

### Как устроено

**Children** — малые чанки, по которым идёт поиск:
- Строятся через MHTS(parent.text) → RCTS(800, 300)
- Хранят `parent_id` — ссылку на родителя
- В `page_content` добавляется упрощённый prefix (`Файл / Тип / Теги`) — участвует в эмбеддинге и BM25

**Parents** — крупные чанки, которые получает LLM:
- Строятся через `RecursiveCharacterTextSplitter(parent_chunk_size=2000, parent_chunk_overlap=200)`
- Хранятся без prefix'а: prefix строится на лету в `retriever/formatting.py`
- Overlap=200 гарантирует: пограничный текст между соседними parents попадёт в children обоих — поиск вернёт два parent'а, LLM увидит обе стороны границы

**Коллекция** — одна, `obsidian_notes`. Разделение через `metadata.kind = "child" | "parent"`.

### Построение parents (итерация 2)

Первоначально parents строились жадной группировкой MHTS-секций (`_build_parents()`): секции объединялись, пока сумма не превысит 2000. Это давало нестабильные результаты — многие parents оказывались значительно меньше 2000 символов (секция 800 → флаш, следующая 1500 в отдельный parent).

Заменено на прямой `RCTS.split_text(body)` с overlap:
- Убрана функция `_build_parents()` (~70 строк)
- `SEPARATORS` расширен: заголовки `\n# `, `\n## `, `\n### `, `\n#### ` добавлены первыми — RCTS режет по границам секций естественно
- `parent_chunk_overlap: 200` добавлен в `IngestConfig` и `config.yaml`
- `parent.heading_hierarchy = []` (не нужен — в prefix'е только `file_name / created / type / tags`)

### Формат prefix'а для LLM (`retriever/formatting.py`)

Строится на лету из `ChunkMetadata`, не хранится в `page_content`:

```
[N]
source: Галаева Елена.md
created: 2025-11-14
source_part: 2/3
type: employee
tegs: команда, HR
---
[текст parent'а]
```

### Поиск

`retriever/search.py` разбит на два этапа:
1. `_search_children()` — гибридный поиск (dense + BM25 + RRF) **только по children** (`filter: metadata.kind = "child"`), опционально реранкер
2. `_dedup_and_fetch_parents()` — дедупликация по `parent_id`, получение parents через `client.retrieve(ids=...)`, score = max child-score для данного parent'а

Публичный API:
- `search(query)` → список parents (для агента)
- `search_with_detail(query)` → `(parents, children)` (для eval и дебага)

### Eval

`eval/runner.py` обновлён: `run_golden_set()` принимает `search_detail_fn` (возвращает кортеж). `EvalDataset` хранит `chunks_detail` (parents) и `chunks_detail_children` (children). `eval/report.py` выводит две таблицы на кейс.

---

## Последствия

- Средний parent: ~1100–1700 символов (против ~800 у старых чанков)
- Overlap=200 решает проблему «информации на границе» без дополнительной логики
- Индекс вырос: каждый файл хранит и children и parents. Оценочно +50–70% к объёму
- Children с overlap из соседних parents создают небольшое дублирование в индексе (~10–15%) — при поиске это не проблема (dedup по `parent_id`)
- Потеряли `heading_hierarchy` у parents — только у children. Это приемлемо: prefix для LLM не использует иерархию заголовков
- Требуется **полная переиндексация** (`python -m retriever.indexer --force`) — chunk_id изменился (добавлен `kind`-префикс в MD5)
- `_build_parents()` удалена полностью — код chunker'а сократился на ~70 строк
