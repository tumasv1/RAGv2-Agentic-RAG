## ADR-0005: Архитектура LangGraph-агента

**Статус:** принято

**Дата:** 15.04.26

---

## Контекст

Нужно реализовать RAG-агента на LangGraph, который:
- решает, нужен ли поиск по базе знаний для данного вопроса
- вызывает инструменты (search, date, hub) и интерпретирует результаты
- формирует финальный ответ только на основе найденных данных
- соблюдает guardrail на 5 итераций
- хранит историю разговора в рамках сессии (short-term memory)

Высокоуровневый план (ADR-0001) описывал 4-нодовый граф:
`analyze → agent ↔ tool_node → generate`. При реализации выявился ряд
вопросов, потребовавших архитектурных решений.

---

## Решение 1: Топология графа — 3 ноды вместо 4

### Варианты

**Вариант А (по плану):** `analyze_node → agent ↔ tools → generate`
- `analyze_node`: отдельный LLM-вызов, который решает "нужен ли поиск?"
- Плюсы: явная логика анализа, легко отлаживать изолированно
- Минусы: лишний LLM-вызов на каждый запрос (+1–2 сек, +стоимость); `agent_node` с `bind_tools()` уже принимает то же решение через system prompt — получается дублирование

**Вариант Б (принято):** `agent ↔ tools → generate`
- `agent_node` с `bind_tools()` и system prompt сам решает: вызвать инструмент / уточнить / ответить
- Плюсы: на один LLM-вызов меньше; логика не дублируется; стандартный ReAct-паттерн
- Минусы: менее явная структура — логика анализа «скрыта» в system prompt

### Решение

Выбираем **Вариант Б**. `bind_tools()` уже включает анализ: если модель
решает не вызывать ни один инструмент — это и есть "поиск не нужен".
Отдельная `analyze_node` дублирует это решение и удваивает задержку.

```
START → agent ──(tool_calls)──→ tools → agent (loop)
                │                        │
                │                 (iteration >= 5)
                │                        │
                └──(no tool_calls)──→ generate → END
```

## Решение 2: generate_node как условный шаг

### Проблема

После реализации выявился дефект: `generate_node` всегда вызывает LLM
заново — даже когда `agent_node` уже дал полноценный ответ. Это приводит
к тому, что хороший ответ агента (message N) перезаписывался шаблонным
ответом `generate_node` (message N+1).

Отладочный пример:
```
[4] AIMessage: "Галаева — это Елена, внутренний аналитик... Источники: ..."
[5] AIMessage: "Если у вас есть вопросы, я готов помочь!"  ← generate_node
```
`ask()` берёт `messages[-1]` → теряется правильный ответ.

### Варианты

**Вариант А:** всегда запускать generate_node с GENERATE_PROMPT
- Плюсы: контроль формата ответа в одном месте
- Минусы: лишний LLM-вызов; модель "переотвечает" поверх уже готового ответа

**Вариант Б (принято):** generate_node пропускает LLM-вызов, если агент уже ответил
```python
last_ai = [msg for msg in reversed(messages) if isinstance(msg, AIMessage)][0]
if last_ai.content and not last_ai.tool_calls:
    return {"messages": []}  # ничего не добавляем
```
- Плюсы: экономия LLM-вызова; ответ агента не перезаписывается
- Минусы: generate_node срабатывает только при лимите итераций или пустом ответе — нужно убедиться, что system prompt достаточно хорошо задаёт формат

**Вариант В:** убрать generate_node совсем
- Плюсы: проще граф
- Минусы: нет запасного LLM-вызова на случай когда агент не ответил (лимит итераций)

### Решение

Выбираем **Вариант Б**. `generate_node` остаётся как "заглушка на крайний случай":
срабатывает только если `agent_node` не дал ответа (например, при лимите итераций
граф пришёл в `generate` минуя обычный путь через `tools_condition`).

## Решение 3: bm25_terms как параметр инструмента, не внутренний LLM-вызов

### Контекст

`retriever/search.py` принимает `bm25_terms: str | None` — жёсткие термины
для BM25 (даты, аббревиатуры, имена). Нужно понять, кто и как их извлекает.

### Варианты

**Вариант А:** tool внутренне вызывает LLM для извлечения терминов
```python
def search_knowledge_base(query: str) -> str:
    bm25_terms = _extract_terms_via_llm(query)  # скрытый LLM-вызов
    return search(query, bm25_terms)
```
- Плюсы: интерфейс проще (один параметр)
- Минусы: +1 скрытый LLM-вызов на каждый поиск; непрозрачно для логов

**Вариант Б (принято):** агент передаёт `bm25_terms` как явный параметр
```python
def search_knowledge_base(query: str, bm25_terms: str | None = None) -> str:
    return search(query, bm25_terms)
```
- Агент видит оба параметра, system prompt объясняет когда и что передавать
- Плюсы: нет скрытых LLM-вызовов; параметр виден в логах tool call; совпадает с API `search()`
- Минусы: агент должен сам извлекать термины — нужны хорошие инструкции в system prompt

### Решение

Выбираем **Вариант Б**. Агент направляется system prompt:

> bm25_terms: извлеки «жёсткие термины» — даты, аббревиатуры, коды, имена собственные.
> Если таких терминов нет — не передавай этот параметр.

Это прозрачно (термины видны в логах как аргументы tool call), экономит LLM-вызов
и согласуется с уже существующим интерфейсом `search()`.

## Решение 4: iteration_count в state рядом с recursion_limit

### Контекст

LangGraph имеет встроенный `recursion_limit` для защиты от бесконечных циклов.
Нужно решить, как реализовать guardrail на 5 итераций.

### Варианты

**Вариант А:** только `recursion_limit`
- Плюсы: ничего не нужно реализовывать самому
- Минусы: при превышении LangGraph бросает `RecursionError` — пользователь видит техническую ошибку вместо понятного сообщения; лимит считается в рёбрах графа, не в итерациях инструментов

**Вариант Б (принято):** `iteration_count` в state + `recursion_limit` как страховка
- `iteration_count` инкрементируется в `tool_node_with_counter`
- `check_iteration_limit` проверяет его после каждого вызова инструментов
- При превышении → `generate_node` формирует ответ на основе накопленного
- `recursion_limit = max_iterations * 2 + 5` — страховка от ошибок в routing
- Плюсы: пользователь получает понятный ответ; счётчик считает именно итерации tool_node
- Минусы: чуть сложнее state

### Решение

Выбираем **Вариант Б**. Явный счётчик даёт контроль над user experience при превышении лимита.

---

## Последствия

- Граф: 3 ноды (`agent`, `tools`, `generate`) — проще, чем 4 по плану
- Обычный путь: 2 LLM-вызова (agent → tools → agent → generate_node/skip)
- generate_node добавляет 3-й LLM-вызов только при лимите итераций
- bm25_terms прозрачны в логах — видно, что именно агент посчитал "жёстким термином"
- При лимите итераций пользователь получает осмысленный ответ, не техническую ошибку
- **Следующий шаг (Batch 2):** per-source лимит в результатах поиска, форматтер контекста для LLM
