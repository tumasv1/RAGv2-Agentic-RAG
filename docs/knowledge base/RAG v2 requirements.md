## Цель

Личный помощник-агент для ответов на вопросы, навигации по базе знаний Obsidian и управления заметками. Доступен через веб-приложение и Telegram бот.

---

## 1. Агент

### 1.1 Поведение

- Перед поиском агент определяет, нужен ли поиск по базе знаний для данного вопроса.
- После получения чанков агент оценивает, достаточно ли их для ответа. Если нет — делает уточняющий поиск.
- Если вопрос непонятен — задаёт пользователю уточняющий вопрос перед поиском.
- Финальный ответ строится только на информации из базы знаний и инструментов. Общие знания LLM в ответ не включаются.
- При ответе всегда указываются источники (названия заметок).
- При ошибке (LLM недоступен, база знаний пуста, поиск ничего не нашёл) агент сообщает об этом пользователю в ответном сообщении.

### 1.2 Guardrails

- Максимум 5 итераций на один запрос. Если лимит превышен — агент останавливается и сообщает пользователю.
- Human-in-the-Loop обязателен при создании, редактировании и удалении заметок.

### 1.3 Память

- Short-term memory: история текущего разговора. Агент отвечает с учётом предыдущих сообщений в сессии.
- Для экономии токенов использовать суммаризацию истории через `summarize_messages()` — длинная история сжимается в краткое резюме.

### 1.4 Реализация

- Реализовать RAG-агента на LangGraph:
- Подключить Langfuse для логирования (бесплатный self-hosted или cloud)
- Все импорты из актуальных пакетов LangChain 1.0: `langchain_core`, `langchain_community`, `langgraph` и т.д. Избегать устаревших импортов из корневого `langchain`.

## 2. Ретривер и поиск

### 2.1 Поиск

- Гибридный поиск: embedding + BM25.
- Reranker для повышения точности. Должен работать быстро на слабом CPU, либо использоваться через API.
- В LLM передаются только чанки выше порогового score. Максимальное количество чанков в контексте ограничено настройкой.
- При составлении контекста использовать имя документа — оно несёт смысловую нагрузку.

### 2.2 Метаданные для ранжирования

- Учитывать поля frontMatter при оценке релевантности:
  - `created` — более свежая заметка обычно актуальнее похожей
  - `type` — помогает понять контекст: задачи, встречи, здоровье и т.д.

### 2.3 Knowledge Graph

- Строить граф из внутренних ссылок `[[...]]` между заметками.
- Использовать граф для расширения поиска: если чанк релевантен, связанные заметки тоже могут быть полезны.

### 2.4 Стратегия чанкинга

Попробовать несколько вариантов сплиттеров и выбрать лучший по метрикам RAGAS:

- `MarkdownHeaderTextSplitter` + `RecursiveCharacterTextSplitter` (и их комбинация)
- `split_obsidian_documents`
- Parent-Child Chunks: поиск по маленьким чанкам (точнее по embedding), в контекст LLM передаётся большой родительский чанк (больше информации).

Сравнение сплиттеров автоматизировать: один скрипт запускает все варианты и выводит метрики RAGAS.

## 3. Схема данных чанка

Минимально необходимые поля каждого чанка:

| Поле | Описание |
|------|----------|
| `chunk_id` | Уникальный идентификатор чанка |
| `parent_id` | ID родительского чанка (для Parent-Child стратегии) |
| `file_path` | Абсолютный путь к файлу |
| `file_name` | Имя файла без пути (передаётся в контекст LLM) |
| `section_header` | Заголовок секции, к которой относится чанк |
| `heading_hierarchy` | Иерархия заголовков `[H1, H2, H3]` — хлебные крошки |
| `type` | Из frontMatter: тип заметки (`project`, `other`, `medical_analysis`, ...) |
| `created` | Из frontMatter: дата создания заметки |
| `tags` | Из frontMatter: теги |
| `extra_metadata` | Все остальные поля frontMatter в виде словаря (произвольные поля) |

> Поля frontMatter нестабильны: одни заметки содержат стандартные поля (`type`, `created`, `tags`), другие — произвольные (`Статус`, `Инициатор`, `analysis_name`). Поэтому нестандартные поля собираются в `extra_metadata`, чтобы не терять информацию.

## 4. Индексирование

- Инкрементальная переиндексация раз в сутки, ночью. Обрабатываются только изменённые файлы.
- Возможность запуска Инкрементальная переиндексация  вручную, через бота или веб-приложение.
- Embedding-модель: `BAAI/bge-m3` или `intfloat/multilingual-e5-large`, запускается локально.
- После каждой переиндексации в базе знаний создаётся заметка-журнал (`.md`) с результатами: сколько файлов обработано, добавлено, удалено, время работы, ошибки.

## 5. Управление конфигурацией

Три варианта на выбор:

**Вариант А (рекомендуется для старта):** `config.yaml` для параметров агента + `.env` для секретов (API-ключи, токены). Просто и прозрачно. Изменения применяются после рестарта.

**Вариант Б:** `config.yaml` + возможность переопределить параметры через веб-UI (UI сохраняет значения обратно в YAML или отдельную БД). Без рестарта, но сложнее в реализации.

**Вариант В:** Только `.env`. Стандарт для Docker-окружений, но неудобен для частой смены параметров агента.

Параметры, которые должны быть настраиваемы:

- Модель LLM и температура
- Максимальное количество чанков в контексте LLM
- Пороговый score для фильтрации чанков
- Максимальное число итераций агента
- Путь к базе знаний Obsidian

## 8. Оценка качества

- Оценка через RAGAS: faithfulness, completeness, accuracy, precision, recall.
- Отдельный скрипт запускает оценку по golden set.
- Golden set: целевой размер — 20 пар вопрос-ответ. Пополняется вручную или через генератор синтетических тест-кейсов.  Часть уже собрана тут generate_golden_set.py
- Сравнение сплиттеров автоматизировано и оценивается через RAGAS.
- Reranker оценивается и настраивается по результатам RAGAS.

import json
from pathlib import Path

def expand_golden_set(notes_dir: str, output_file: str, n_per_doc: int = 3):
    """Генерирует тест-кейсы из документов в базе знаний."""
    all_cases = []

    for md_file in Path(notes_dir).glob("*.md"):
        text = md_file.read_text(encoding="utf-8")
        if len(text) < 200:  # пропускаем слишком короткие файлы
            continue

        cases = generate_test_cases(text, n=n_per_doc)
        for case in cases:
            case["source"] = md_file.name
            all_cases.append(case)

    with open(output_file, "w", encoding="utf-8") as f:
        json.dump(all_cases, f, ensure_ascii=False, indent=2)

    print(f"Сгенерировано {len(all_cases)} тест-кейсов")
    return all_cases

### Пример реализации на langgraph 
#### Subgraphs — модульность

Большой агент можно разбить на подграфы:

```python
# Подграф для retrieval
retrieval_builder = StateGraph(RetrievalState)
retrieval_builder.add_node("search", search_node)
retrieval_builder.add_node("rerank", rerank_node)
retrieval_builder.add_edge("search", "rerank")
retrieval_subgraph = retrieval_builder.compile()

# Главный граф использует подграф как node
main_builder = StateGraph(AgentState)
main_builder.add_node("retrieve", retrieval_subgraph)  # целый подграф = один node!
main_builder.add_node("generate", generate_node)
main_builder.add_edge("retrieve", "generate")
```

#### Retry / Fallback

```python
def should_continue_with_retry(state: AgentState) -> str:
    last_message = state["messages"][-1]

    # Tool вернул ошибку → retry
    if hasattr(last_message, "content") and "Error" in str(last_message.content):
        if state.get("retries", 0) < 3:
            return "retry"
        return "fallback"

    # Результатов мало → reformulate query
    if state.get("docs_count", 10) < 2:
        return "reformulate"

    return "generate"

# Добавляем retry и fallback nodes
builder.add_node("retry", retry_node)
builder.add_node("fallback", fallback_node)
builder.add_node("reformulate", reformulate_query_node)

builder.add_conditional_edges("check_results", should_continue_with_retry, {
    "retry": "call_tools",
    "reformulate": "reformulate",
    "fallback": "fallback",
    "generate": "generate"
})
```
